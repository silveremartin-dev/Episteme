3. Hoped:

Next version should focus on:
1. Testing and debugging
2. Documentation
3. XML internal data io support
4. internationalization
5. parametrization (merge with Jean Marie Dautelle)
6. some packages changes depending on the requests
7. especially change the model of space and time to provide complete units support (with minimal performance overhead)
8. more separation of gui from the rest of the API
9. more interfaces and modular oriented behavior

Said in short the next milestone (version 2.0) should involve a tigher API with less overlapping between packages, loaders and gui should be more separated (MVC model enhanced), XML datafiles should be transparently loaded and saved as part of the API (datafeed), more subpackages should be merged in (for example in physics). Also, we have to think about something powerful for coordinates as we use different systems for geometry, geography and physics. All this is work but the real difficult part would be to go JSR.

4. JSR "go to" status

First we should send a mail to Sun to insist on the fact that science is poorly managed in Java and that we have done a huge amount of work to clear the path. Then we should urge them that there is potential with a JSR.

Though I haven't much changes to suggest upon the whole API, XML parsers should be used in their corresponding packages (a lot of changes and work here), better coding style more consistent bewteen packages, a real MVC model with viewers really rewritten to use the underlying packages, a complete rewrite of bibliography using a real standard (that I am not aware of) (although this API is very useful, it really looks pittyful, IMVHO).

Related: io standard formats should be considered, more devices drivers added. Viewers subpackages should not be part of the JSR API and be really rewritten to integrate really with the underlying API. As it is now done, the library should be delivered with public domain datafiles (periodic table...) and XML enhancements should be done in this area to provide better support by the API itself. Here is an example, imagine a timeline of world events for history: the whole sequence would be the following a data file is parsed by a specific XML parser and then passed to the history package and then viewed with corresponding swing timeline viewer. Epistemepackages.zip provides all these functionnalities but with a level less integration: users have to manually build bridges between packages.
